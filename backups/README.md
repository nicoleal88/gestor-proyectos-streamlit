# Backup System for Gestor de Proyectos

This directory contains the backup system for the SQLite database used by the Gestor de Proyectos application.

## Components

- `backup_db.py`: Main backup script that creates local and remote (SSH) backups.
- `backup_status.json`: Status file generated by the backup script for the frontend to display backup status.
- `backup.log`: Log file for the backup script.
- `gestor_*.db`: Timestamped backup files.

## Configuration

Backup behavior is controlled by environment variables, which can be set in the `.env` file at the project root:

| Variable | Description | Default |
|----------|-------------|---------|
| `DATABASE_PATH` | Path to the SQLite database file | `data/gestor.db` |
| `BACKUP_DIR` | Directory where backups are stored | `backups` |
| `BACKUP_REMOTE_SERVERS` | Space-separated list of remote servers in the format `user@host:/path/to/backups/` | (empty) |
| `BACKUP_SSH_KEY` | Path to the SSH private key for remote backups | `~/.ssh/id_ed25519` |
| `BACKUP_RETENTION_DAYS` | Number of days to retain backups (local only) | `7` |

### Example `.env` Configuration

```env
DATABASE_PATH=data/gestor.db
BACKUP_DIR=backups
BACKUP_REMOTE_SERVERS="backupuser@backup1.example.com:/backups/gestor/ backupuser@backup2.example.com:/backups/gestor/"
BACKUP_SSH_KEY=/home/nleal/.ssh/id_ed25519_backup
BACKUP_RETENTION_DAYS=7
```

## SSH Setup for Remote Backups

To enable backups to remote servers via SSH:

1. **Generate an SSH key pair** (if you don't already have one dedicated for backups):
   ```bash
   ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_backup -N ""
   ```
   This creates a key without a passphrase for automation.

2. **Copy the public key to the remote server(s)**:
   ```bash
   ssh-copy-id -i ~/.ssh/id_ed25519_backup.pub backupuser@backup1.example.com
   ssh-copy-id -i ~/.ssh/id_ed25519_backup.pub backupuser@backup2.example.com
   ```
   Ensure the user has write permissions to the backup directory on the remote server.

3. **Test the SSH connection**:
   ```bash
   ssh -i ~/.ssh/id_ed25519_backup backupuser@backup1.example.com "echo 'SSH connection successful'"
   ```

4. **Update the `.env` file** with the `BACKUP_REMOTE_SERVERS` and `BACKUP_SSH_KEY` variables.

## Manual Testing

To test the backup script manually:

```bash
# Load environment variables from .env
export $(grep -v '^#' .env | xargs)
# Run the backup script
python3 backups/backup_db.py
```

Check the output and the generated `backup_status.json` file.

## Automated Backups with Cron

To run the backup script daily at 02:30 AM, add the following line to the crontab (run `crontab -e`):

```cron
30 2 * * * cd /home/nleal/GitHub/gestor_proyectos_streamlit && export $(grep -v '^#' .env | xargs) && python3 backups/backup_db.py >> backups/backup_cron.log 2>&1
```

This will:
- Change to the project directory
- Load environment variables from `.env`
- Run the backup script
- Append both stdout and stderr to `backups/backup_cron.log`

### Log Rotation

Consider setting up log rotation for `backup.log` and `backup_cron.log` to prevent excessive disk usage. You can use `logrotate` or a simple cron job to truncate logs weekly.

## Restoring from Backup

To restore the database from a backup:

1. Stop the application (if possible) to avoid conflicts.
2. Copy the desired backup file to the database location:
   ```bash
   cp backups/gestor_YYYYMMDD_HHMMSS.db data/gestor.db
   ```
3. Restart the application.

## Troubleshooting

- **Backup fails silently**: Check `backups/backup.log` for detailed error messages.
- **SSH connection issues**: Verify the SSH key and remote server accessibility manually.
- **Status not updating in UI**: Ensure the `backup_status.json` file is being written and is valid JSON.
- **Disk space full**: The script retains only the last 7 days of backups locally, but remote servers may need their own cleanup policy.

## Notes

- The backup script uses SQLite's online backup API to create a consistent backup without locking the database for writes.
- Remote backups are attempted only if `BACKUP_REMOTE_SERVERS` is set and non-empty.
- The status file is written atomically to avoid partial reads by the frontend.
